# Obsidian知识库体检：一条Prompt让Claude Code找出所有问题

## 开场钩子

你的 Obsidian 知识库里，**60% 的笔记可能是没人引用的「孤岛」**——而你自己完全不知道。

我们搭知识库时，一开始架构总是设计得很精致。但用着用着，它就会一点一点偏离最初的样子：层级越来越多、标签越来越乱、命名越来越随意。最后变成一个「看起来内容丰富，实则一盘散沙」的仓库——更可怕的是，你对此毫无察觉。

这时候，你需要一个**外部视角**。一个能同时扫一眼上百篇笔记、统计链接密度、找出命名冲突、标记重复与矛盾，然后清楚地告诉你「这里有问题」的工具。

今天分享一条 Prompt。丢给 Claude Code，一键体检你的整个知识库。

## 附：完整 Prompt

> 你现在是我的Obsidian知识库资深架构师和重构专家，已帮助多人将混乱的个人知识库转化为高效、连贯的第二大脑。你熟悉Obsidian最佳实践（PARA、Zettelkasten、Johnny Decimal、Atomic Notes、LATCH组织法等），也了解常见痛点：文件夹层级过深/过浅、笔记孤岛（orphans）、重复内容、命名不一致、链接稀疏、MOC缺失、原子性不足等。
>
> 我的知识库已经运营一段时间，现在请你对整个vault进行全盘审查。vault路径/文件夹已提供给你（或通过工具访问），请全面分析。

### 审查重点（必须覆盖）

**1. 整体结构评估：**

- 当前文件夹结构和分类逻辑的优缺点。
- 是否存在不合理的层级（过深导致导航困难，或过浅导致混乱）。
- 推荐的理想文件夹/分类方案（可参考PARA + 项目专区 + 知识领域 + 资源库等），并解释理由。

**2. 分散与碎片化问题（Scattered notes）：**

- 识别主题/概念分散在多个地方的笔记（相同或相似内容未聚合）。
- 建议如何合并、链接或创建MOC（Map of Content）/Hub笔记来集中它们。
- 查找"几乎相同"或重叠的笔记（duplicates或near-duplicates）。

**3. 不合理的地方（Unreasonable/Inconsistent）：**

- 命名约定不一致（标题、文件名前缀、日期格式等）。
- 前言（Frontmatter）使用混乱或缺失关键元数据（tags、aliases、status等）。
- 原子性问题：笔记过长/过杂（应拆分）或过短/无价值（应合并/删除）。
- 链接质量：死链、弱链接、缺少双向链接。
- 孤岛笔记（orphans）和死胡同（无出链/入链的笔记）。
- 工作流断点：如Inbox堆积、未处理的Daily Notes、未归档内容。

**4. 知识连贯性与可发现性：**

- 核心知识图谱是否连贯？哪些领域有明显空白或冗余？
- 搜索/导航效率：推荐的索引、Dashboard、查询或模板改进。
- 长期维护性：哪些习惯会导致未来再次分散？

**5. 量化洞察（如果工具支持统计）：**

- 笔记总数、文件夹分布、平均链接数、未链接笔记比例等。
- 高价值 vs 低价值笔记的识别。

### 输出格式要求（严格遵守，便于我行动）

- **执行摘要**：3-5句总体诊断（最严重的问题是什么？）。
- **优势**：目前做得好的地方（保留并强化）。
- **主要问题**：按严重程度排序，每点列出具体例子（笔记路径/标题）和影响。
- **重构建议**：具体、可执行的行动计划（分优先级：立即做 / 本周 / 长期）。包括新文件夹结构提案、批量处理命令或Dataview查询、命名规则模板等。
- **MOC/Hub推荐**：建议创建哪些枢纽笔记，以及如何迁移内容。
- **下一步Prompt**：提供1-2个后续prompt，让我能让你分批执行重构（例如先处理某个文件夹）。
- **可选**：如果vault很大，建议分阶段审查策略。

> 请保持直接、坦诚、专业，不要过度礼貌软化批评。基于事实和最佳实践说话。如果需要更多具体笔记内容或统计，请告诉我，我会提供。开始审查吧。

## 解构这条 Prompt 的四个设计点

这条 Prompt 为什么能输出如此高质量的审查报告？拆成四个核心设计点。

### 设计点一：角色定义 + 专业坐标系

不是简单让 AI「扮演专家」，而是给它画了一个清晰的专业坐标系。明确告诉 AI：你是「Obsidian 知识库资深架构师和重构专家」，同时植入 PARA、Zettelkasten、Atomic Notes、Johnny Decimal 等最佳实践框架。这让 AI 从一开始就站在专业高度审视你的知识库，而不是用通用知识随便应付。

### 设计点二：强制定量分析 + 点名批评

没有让 AI 泛泛而谈。强制列出了 5 大审查重点——整体结构、碎片化、命名规范、链接质量、知识连贯性——每一项都必须逐条检查。同时要求「按严重程度排序」，给出**具体笔记路径和例子**。逼着 AI 不能含糊其辞，必须点名批评。

### 设计点三：禁止讨好

整条 Prompt 里最狠的一招。最后一句特别强调：**「请保持直接、坦诚、专业，不要过度礼貌软化批评。」**

这句话直接堵死了 AI 的「好好先生」模式。没有它，AI 会把严重问题包装成「小建议」。有了它，AI 才敢说「60% 是孤岛」「根目录正在变成垃圾场」这种直白又扎心的实话。这就是**好 Prompt 和普通 Prompt 的分水岭**。

### 设计点四：内置接力棒

这条 Prompt 最聪明的地方：**不只是诊断，还自带执行方案**。审查报告末尾会自动生成 1-2 个后续 Prompt，让你能继续让 AI 分批执行重构，避免一次性信息过载。Vault 很大的话，还会给出分阶段审查策略——先审 Daily 文件夹，修完再审 Script，稳扎稳打。

## 记住三点就够了

**① 雪盲效应** —— 你对自己 Vault 的混乱已经「雪盲」了。那些冗余、孤岛、矛盾，在你眼里早就变成了背景。你看不见的问题，AI 能一眼看出来。它就是你的外部审计视角——专业、客观、毫不留情。

**② 降低启动摩擦** —— 大多数知识库不是死于没有方法，而是死于「一直没开始」。这条 Prompt 把启动摩擦降到了最低：复制、粘贴、回车。第一步就这么简单。

**③ 不挑规模** —— 100 篇还是 1000 篇都能用。量大就分批执行，一个文件夹一个文件夹来，边修边推进。

## 使用边界，说清楚

**工具要求：** 需要 Claude Code 的 Agent / Computer Use 能力（能读文件、统计、写文件）。普通 ChatGPT 网页版或基础模型只能给建议，无法真正执行审查——不是能力问题，是缺了读写文件这个物理条件。

**不是「一键魔法」：** AI 有时会建议合并两个「看起来重复」的笔记，但你可能有自己的理由保留它们。拿到报告后，**一定自己过一遍再执行**。

**Token 消耗：** 实测纯审计阶段约 30-40K token，执行阶段取决于修改量。整体属于正常对话消耗。
